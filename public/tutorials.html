常用教程都放在这里。遇到“世界书导入后没反应”“多端同步怎么配”“ComfyUI 后端怎么启动”“GPT Image2 怎么用 API Key 生图”“预设图怎么反馈”“DeepSeek 模式怎么选”等问题，可以直接切换下面的选项卡查看。

怎么导入自己的世界书

  1. 回到首页，点击 世界书管理。
  2. 切换到 附加世界书，点击左侧 导入，选择自己的 JSON 文件。
  3. 导入后检查左侧是否出现新世界书，右侧是否有条目内容。
  4. 确认 启用这本附加世界书 是打开状态，条目内容不为空。
  5. 新手建议把条目设置为 始终注入，作用域勾选 全部流程。
  6. 最后一定要点右上角 保存全部。只导入不保存，刷新或重开后可能看起来像没生效。

为什么添加了没反应

常见原因不是坏了，而是没有满足注入条件。请逐项检查：世界书启用、条目启用、条目内容不为空、作用域包含当前流程、导入后点过保存。

如果条目是 关键词命中，必须填写关键词，并且当前剧情、地点、NPC 或最近历史里出现关键词才会注入。想稳定生效请先改成 始终注入。

支持哪些格式

支持本项目导出的 extra-worldbooks.json，也兼容 SillyTavern / 酒馆 World Info JSON。标准格式通常是顶层 entries 对象，条目里包含 key、comment、content、constant、disable 等字段。

注意：常见 SillyTavern 角色卡是 PNG 图片或角色 JSON，不等同于世界书。当前入口用于导入世界书 JSON；如果拿到的是角色卡 PNG，请先在酒馆里导出/整理对应 World Info JSON。

世界书不会直接改存档变量，也不会马上改变当前画面。它是在下一次生成正文、开局、变量生成、世界演变等请求时，作为提示词补充交给模型。

主剧情消息模式怎么选

主剧情消息模式决定“玩家本回合输入”和“游戏提示词”用什么方式发给 AI。不同模型的习惯不一样：Gemini 更适合原本的组包方式，GPT 系列更喜欢最后一条是真实 user，DeepSeek 则更容易遇到 reasoning 泄漏、格式漂移或标签缺失，所以有专门的稳定策略。

如果你不知道该选什么：Gemini 选 Gemini模式，OpenAI/GPT 兼容站选 GPT兼容，DeepSeek 先选 DeepSeek标准续聊，格式老坏再试 DeepSeek锁格式。

Gemini模式

这是项目原本的默认组包方式，最适合 Gemini 系列模型。它会按游戏原有提示词结构组织上下文，不启用 DeepSeek 的 Prefix、锁格式和健康度策略。

什么时候用

  1. 你使用 Gemini、Gemini Flash、Gemini Pro 等模型。
  2. 你刚开始玩，不确定要选哪个模式。
  3. 当前生成正常，没有标签缺失、正文跑偏等问题。

普通玩家首选这个。不是 DeepSeek 模型时，不建议为了“看起来高级”去开 DeepSeek 模式。

GPT兼容

GPT兼容不是摆设，它会真正改变主剧情请求方式：最后一条消息会直接使用玩家本回合输入，让请求更接近常见 Chat Completions 对话格式。

什么时候用

  1. 你使用 OpenAI、GPT、GPT 兼容接口或某些第三方 OpenAI 兼容站。
  2. Gemini模式下模型不理解“开始任务”，或者经常忽略本回合玩家输入。
  3. 接口要求最后一条消息必须是 user。

GPT兼容没有 DeepSeek 专属 Prefix 或锁格式策略。它只是把主剧情消息结构改得更适合 GPT 类接口。

DeepSeek标准续聊

这是 DeepSeek 推荐优先尝试的模式。它会使用真实玩家输入作为最后一条 user，默认关闭 COT 伪装注入，并注入 DeepSeek 专用稳定提示词，减少续聊污染和 reasoning 泄漏。

什么时候用

  1. 你使用 DeepSeek Chat、DeepSeek Reasoner 或 DeepSeek 兼容接口。
  2. 你不确定当前接口是否支持 Prefix/Prefill。
  3. 你希望先保证兼容性，不想一上来就用实验性锁格式。

DeepSeek 玩家建议先用标准续聊。如果正文正常、标签完整，就不用切锁格式。

DeepSeek锁格式

锁格式是 DeepSeek 的强约束模式。它会尝试用 Prefix/Prefill 让模型从固定标签开头继续写，从而减少“忘记输出标签”“乱加说明”“Markdown 包裹结果”等问题。

什么时候用

  1. DeepSeek标准续聊经常缺少 <正文>、<短期记忆> 或 <命令>。
  2. 模型内容能写好，但格式经常漂移。
  3. 你使用 DeepSeek 原生接口，或确认第三方兼容站支持 Prefix/Prefill。

锁格式不是“更聪明”模式，而是“更守格式”模式。如果接口不支持 Prefix，会自动降级为标准续聊；如果文风变硬或接口报错，可以切回标准续聊。

DeepSeek 主线稳定性策略详解

这些选项只在 DeepSeek标准续聊 或 DeepSeek锁格式 下显示。新手可以保持默认；遇到格式问题时，再按下面说明调整。

开局策略

开局策略控制第 0 回合，也就是新存档第一段剧情，是否直接使用 DeepSeek 的稳定模式。

选项解释

  1. 禁止开局，稳定模型起盘后接管：不让 DeepSeek 负责最复杂的开局起盘，适合追求稳定的新手。
  2. 标准开局：开局也使用 DeepSeek 标准续聊方式。
  3. 锁头开局（实验）：开局也尝试锁格式 Prefix，适合排查开局格式老坏的问题。

推荐默认使用“禁止开局”。开局要初始化角色、世界、第一幕，任务比普通续聊更复杂，稳定优先。

接管摘要

接管摘要可以理解为“给后续救场和上下文清理准备一份干净摘要”。当 DeepSeek 输出被 reasoning、格式错误或多余解释污染时，系统可以用摘要帮助后续继续剧情，而不是把脏内容一直带下去。

什么时候开

  1. 默认建议开启。
  2. 长存档、长上下文、剧情很多时尤其建议开启。
  3. 如果你发现后续回合总被前一轮坏格式影响，也建议开启。

Prefix能力探测

Prefix/Prefill 是“先帮 AI 写好回复开头，让 AI 接着往下写”的能力。锁格式会依赖它来固定输出结构，例如先给模型一个 <thinking> 或固定标签开头，让它不要乱起头。

为什么需要探测

  1. DeepSeek 原生接口通常支持 Prefix，但需要走专门的 beta 接口。
  2. 很多第三方兼容站虽然模型名叫 DeepSeek，但不支持 Prefix 字段。
  3. 如果不探测就强行使用，接口可能直接报错。

建议一直开启。开启后，系统会尝试判断当前接口能不能用 Prefix；不能用时自动降级成标准续聊，避免玩家手动试错。

输出健康度检测

输出健康度检测就是给 AI 本轮回复打一个“格式健康分”。它不是评价文笔好不好，而是检查这次输出能不能被游戏稳定解析。

主要检查什么

  1. 有没有必需标签，例如 <正文>、<短期记忆>、<命令>。
  2. 正文是不是为空、太短或被说明文字污染。
  3. 有没有 Markdown 代码块、道歉、接口说明等不该出现的内容。
  4. 有没有 reasoning/thinking 思考内容泄漏到玩家可见文本里。
  5. 命令段是否明显无法解析。

建议 DeepSeek 用户开启。它可以帮助后续决定是否需要锁格式、重试或救场。

锁格式阈值

锁格式阈值可以理解成“低于多少分就认为格式不够稳，需要更严格”。默认值通常是 85。

怎么调

  1. 默认 85：适合大多数玩家。
  2. 调高：更严格，更容易触发锁格式，适合经常缺标签的情况。
  3. 调低：更宽松，适合只是偶尔小问题、不想频繁触发锁格式的情况。

不要一上来调到 100。太严格可能让系统把本来能用的回复也判成不稳定。

救场阈值

救场阈值是“低于多少分就认为这次输出坏得比较严重，需要救场模型或更强修复”。默认值通常是 60。

怎么调

  1. 默认 60：只在明显坏格式时救场。
  2. 调高：更容易救场，适合 DeepSeek 经常空回复、缺大段标签、命令段无法解析时。
  3. 调低：更少救场，适合想节省 API 调用次数的玩家。

救场通常会多消耗一次模型请求。如果只是偶尔缺一个小标签，优先调整锁格式阈值，不一定要频繁救场。

续聊Thinking / 开局Thinking

DeepSeek Reasoner 这类模型可能输出 reasoning/thinking。Thinking 有时能增强推理，但也可能污染正文、让玩家看到思考过程，或者增加延迟。

推荐设置

  1. 普通玩家建议关闭。
  2. 如果出现“思考内容混进正文”，必须关闭。
  3. 如果你使用 Reasoner，且想让开局或复杂剧情推理更充分，可以尝试开启，但要观察格式是否变差。

开局Thinking只影响第 0 回合；续聊Thinking影响后续正常游玩回合。

DeepSeek稳定救场模型

稳定救场模型在“接口配置中心”里设置。它是备用模型：当 DeepSeek 主模型输出健康度太低时，用另一个模型帮忙修复或重写格式。

怎么填

  1. 如果只填“稳定救场使用模型”，API 地址和密钥留空，会复用当前主接口。
  2. 如果你有另一个更稳定的接口，可以单独填写 API 地址和 API 密钥。
  3. 主模型用 deepseek-reasoner 时，救场模型可以先试 deepseek-chat。

救场模型不是必填。只有你经常遇到坏格式、空回复、标签缺失时，再开启它。

玩家快速决策表

  1. 用 Gemini：选择 Gemini模式。
  2. 用 GPT 或普通 OpenAI 兼容站：选择 GPT兼容。
  3. 用 DeepSeek：先选择 DeepSeek标准续聊。
  4. DeepSeek 经常缺标签或乱加说明：改成 DeepSeek锁格式，保持 Prefix能力探测 开启。
  5. DeepSeek 输出经常坏到不能解析：开启 输出健康度检测，必要时配置 稳定救场模型。
  6. 看到思考过程混进正文：关闭 续聊Thinking 和 开局Thinking。

GPT Image2 API 生图

如果你想直接用 API Key 调用 GPT Image2 生图，可以使用自己信任的 API 服务。首页已撤下所有第三方站点入口，不再展示或推荐相关服务。

请只使用自己确认可靠的 API 来源。填写 API Key 前，请先确认服务条款、价格、模型可用性和密钥安全风险。

怎么填到游戏里

  1. 在自己信任的 API 服务后台创建或复制自己的 API Key。
  2. 回到墨色江湖，打开 设置 或 接口配置中心。
  3. 进入 文生图 配置，把 API 地址、模型名称和 API Key 按站点说明填写进去。
  4. 保存后先生成一张测试图，确认额度、模型和图片返回都正常。

适合哪些玩家

  1. 不想自己搭 ComfyUI 后端，只想直接填 Key 使用。
  2. 想用较低成本测试自动 NPC 生图、场景生图或物品图。
  3. 设备不方便长期挂本地生图环境。

如果你已经有稳定的本地 ComfyUI 或 CNB ComfyUI，可以继续使用原方案；API Key 方案更偏向“开箱即用”。

物品预设图反馈

如果你发现某个物品预设图不贴合名称、类型或品质，可以打开反馈页面快速查看全量物品图，并把需要重做的条目记录下来。

打开预设图反馈页

反馈页会展示当前线上物品预设图，适合集中检查“图像是否像这个物品”“是否有文字水印”“是否人物入镜”“同类装备是否风格统一”等问题。

怎么反馈更有效

  1. 打开反馈页后，先按物品类型或名称找到有问题的图。
  2. 记录物品名称，并简单说明问题，例如“不像靴子”“有文字”“像武器不是材料”。
  3. 如果有多个问题图，可以按类型分组反馈，方便下一批集中重做。
  4. 反馈时尽量保留物品名称原文，避免同名简称导致定位错误。

适合检查什么

  1. 武器、防具、秘籍、材料、饰品、杂物是否和名称一致。
  2. 图片是否出现人物、手、界面边框、文字、水印或多物品拼图。
  3. 同一系列物品是否明显跑偏，例如鞋子生成成护腿、地图生成成卷轴以外的杂物。

反馈页只用于查看和整理问题，不会直接修改你的存档。预设图修正会随后续版本统一发布。

多端同步教程

推荐先在一台常用设备上完成云同步配置，再让其它设备登录同一个同步账号并拉取云端存档。

进入主站 进入备用站 下载最新版 APK

推荐方式：对象存储同步

  1. 在原设备打开“云端同步”。
  2. 选择“对象存储同步”，填写用户名、Access Key、Secret Key、Endpoint、Bucket 和同步前缀。
  3. 点击连接测试，通过后先执行“上传到云端”。
  4. 另一台设备填写同一套配置，再执行“从云端恢复”或“下载云端存档”。

没有对象存储账号也可以参考 hi168：无需实名，免费额度约 500G。参考帖：NodeSeek hi168 教程，这个网址需要翻墙。

备用方式：GitHub 云同步

  1. 打开“云端同步”，选择 GitHub 同步。
  2. 按页面提示登录 GitHub 并完成授权。
  3. 在原设备上传云端存档。
  4. 在新设备登录同一个 GitHub 账号，然后从云端恢复。

如何构建自己的 CNB ComfyUI 后端

注册并登录 CNB 后，Fork 指定仓库，进入工作区等待自动启动，再在墨色江湖里填写自己的 CNB 用户名称，即可自动连接自己的 ComfyUI。

本教程只适用于墨色江湖 bacon159 二创版。主入口：https://msjh.bacon159.pp.ua/；备用入口：https://msjh.bacon.de5.net/。

启动步骤

  1. 打开 https://cnb.cool/bacon159-2026/comfyui-ql 并 Fork 到自己账号。
  2. 进入自己的 Fork，点击右上角橙色 启动 L40 预装环境。
  3. 等待终端出现 ComfyUI 8188 ready、Detected ComfyUI URL、initial sync ok。
  4. 保持 CNB 工作区页面打开，用于保活。
  5. 回到墨色江湖“自动发现 ComfyUI 后端”，填写 CNB 个人主页头像下方的用户名称并刷新列表。

用户名称位置

打开 CNB 个人主页，看左侧头像下方的中文用户名称。这个名称就是游戏里的“CNB 用户名称”。

常见启动问题

如果终端最后停在 torch.AcceleratorError: CUDA error: uncorrectable ECC error encountered，通常是 CNB 分配到的 GPU 节点异常。建议关闭当前工作区后重新创建或重启 L40 工作区；仍然报错时，把报错发给 CodeBuddy 或 CNB 反馈。

如果终端明确报错，才尝试手动执行：

cd /workspace && \

CNB 模型贡献教程

如果你发现某个 ComfyUI 工作流需要额外模型，可以把模型名称和下载地址贡献到 CNB 仓库。最简单的做法是：Fork 我的 CNB 仓库，按格式添加“模型名称 + 下载地址”，然后发起合并请求。

适合贡献的内容包括 checkpoint、LoRA、VAE、ControlNet、放大模型、文本编码器、clip、unet、角色或风格专用模型等。不要上传无授权、禁止分发或来源不明的模型。

最简单贡献方式

  1. 打开本地或 CNB 上的 ComfyUI 仓库，找到仓库根目录里的 自定义下载链接.yaml。
  2. 这个文件一行就是一个模型，格式固定为 模型保存路径: 下载链接。
  3. 新增模型记录时，把左边写成模型要保存到 ComfyUI 的路径，右边写成可直接下载的链接。
  4. 下载地址可以是 HuggingFace、ModelScope、网盘直链、CNB 仓库文件链接，或你自己维护的公开模型链接。
  5. Fork 我的 CNB 仓库，把修改提交到自己的 Fork；提交弹窗里可以直接提交到 main，也可以选择创建新分支。
  6. 提交成功后回到仓库首页，点击顶部提示里的 创建合并请求。
  7. 在合并请求说明里写清楚这个模型对应哪个工作流、放到哪个目录、用途是什么。

贡献清单时请尽量保留原始文件名，例如 xxx.safetensors。模型名称和下载后文件名最好一致，这样自动下载脚本更容易定位。

提交修改并创建合并请求

  1. 改完 自定义下载链接.yaml 后，点击页面底部或编辑器里的橙色 提交。
  2. 在提交弹窗里填写简短描述，例如 编辑文件 自定义下载链接.yaml；一般可以选择 直接提交到 main 分支。
  3. 提交完成后回到仓库代码页，顶部会提示“比上游分支多 1 次提交，可创建合并请求”。点击 创建合并请求。
  4. 合并请求标题建议写清楚模型或工作流名称，说明里补充模型来源、下载链接、保存目录和授权情况。

推荐记录格式

自定义下载链接.yaml 不是 JSON，而是最简单的 YAML 映射：左边是模型路径，右边是下载链接。一个模型一行。

# 一个模型一行，如果模型路径中有空格，需要加英文双引号。自动按sha256或url去重（例如下面的例子，只会添加第一个文件）
# 文件路径可以是绝对路径，也可以是相对于comfyui或comfyui/models的路径，会自动裁剪处理

提高下载速度：用 CNB 仓库存模型

如果模型从外网下载慢，可以自己创建一个公开 CNB 仓库来存模型文件。ComfyUI 运行在 CNB 时，用 CNB 仓库里的文件链接下载，通常会走 CNB 内网，速度会快很多。

CNB 单个用户仓库对象存储上限约 100G。要传的模型总量超过 100G 时，可以分多个 CNB 账号放；一个手机号对应一个微信账号可以注册一个 CNB 账号，可以同一个实名。

创建公开 model 仓库

  1. 进入 CNB 个人主页，点击 代码仓库。
  2. 点击右侧橙色 创建仓库。
  3. 仓库名称填写 model。名字可以随便取，叫 model 只是方便理解。
  4. 公开性选择 公开，确保其他人和 CNB 工作区可以访问下载链接。
  5. 点击 创建。
  6. 进入新仓库后点击 云原生开发，再点击 WebIDE 进入 VSCode 界面。

上传模型文件

  1. 在 WebIDE / VSCode 左侧文件区创建合适目录，例如 checkpoints、loras、vae。
  2. 把本地模型文件直接拖到左侧文件区。
  3. 大模型一定要用 Git LFS 提交，否则普通 Git 很容易失败或仓库异常变大。
  4. 建议一个仓库只放一批相关模型，后续维护和复制链接更清楚。

必须使用 Git LFS 提交

需要安装并启用 Git LFS。先在 WebIDE 终端里执行 git lfs install；如果提示 git: 'lfs' is not a git command 或 git-lfs: command not found，说明当前环境还没装 Git LFS，需要先按 CNB / 镜像环境安装 git-lfs 后再继续。

安装可用后执行下面命令。文件名和目录按你的实际模型路径改。

git lfs install
git lfs track "*.safetensors"
git lfs track "*.ckpt"
git lfs track "*.bin"
git lfs track "*.pth"
git add .gitattributes
git add checkpoints/你的模型.safetensors
git commit -m "add model"
git push origin main

这一步一定需要 Git LFS。不要只用普通 git add 硬传大模型。第一次 push 如果要求登录或授权，按 CNB 页面提示完成即可。

找到模型下载链接

  1. push 成功后，回到 CNB 仓库的 代码 页面。
  2. 点进刚上传的模型文件，例如 checkpoints/你的模型.safetensors。
  3. 在文件页面复制下载链接或原始文件链接。
  4. 把这个链接填回 ComfyUI 仓库的自定义下载链接配置里。
  5. 用这个链接下载时，如果 ComfyUI 也跑在 CNB，通常会走 CNB 内网，速度会明显更快。

PR 时请写清楚

  1. 模型名称和文件名。
  2. 下载地址。
  3. 模型应该放到哪个目录，例如 models/checkpoints 或 models/loras。
  4. 对应哪个 ComfyUI 工作流，是否必需。
  5. 来源说明和授权情况。

如果模型很大，建议先自己用 CNB 工作区测试一次能否下载成功，再发 PR。
