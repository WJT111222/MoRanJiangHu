# 移植分析：物品生图提示词优化

> **分支**: `Rebase` 相对于 `origin/main`  
> **分析日期**: 2026-07-09  
> **涉及提交**: `c400c614` / `37e3ae1d` / `d5736b7b`

---

## 一、改动总览

本次改动引入了 **按后端类型区分提示词策略** 机制：

| 条件 | 策略 |
|------|------|
| 模型名含 `grok` / `imagine` / `gpt` / `image` | 中文描述 + 简短画风约束，跳过负面提示词 |
| 本地 / CNB / SD WebUI / ComfyUI 等 | 保持原有英文描述堆砌逻辑不变 |

核心理念：云端大模型（Grok/GPT）对中文提示词理解能力强，不需要冗长的英文描述堆砌；本地/CNB 模型则需要完整的英文提示词才能产出合格图片。

## 二、文件级改动

### 2.1 `services/ai/itemImageGeneration.ts`

#### 2.1.1 新增导入与接口扩展

| 位置 | 改动 |
|------|------|
| L1-12 | 新增导入 `题材模式类型`、`查询图库工具`、`提取图库物品名称`、`删除用户图库条目`、`读取图片资源`、`创建图片资源引用` |

#### 2.1.2 游戏机制文案过滤放宽

**原逻辑**（origin/main）：

```
游戏机制关键词 = /兑换|强化|支线剧情|奖励点|属性|技能|等级|经验|伤害|生命值|法力值|冷却|暴击|命中|闪避|抗性|穿透|吸血|回蓝|buff|debuff|...
```

宽泛正则，会误伤描述性自然语言。

**新逻辑**（Rebase）：

```typescript
const 是否游戏机制文案 = (text: string): boolean => {
    const 数值模式 = /属性[+\-]\d+|伤害[+\-]\d+|生命值[+\-]\d+|法力值[+\-]\d+|冷却[+\-]?\d+秒|暴击率[+\-]\d+%|.../i;
    const 功能模式 = /承载.*支线剧情|用于兑换|兑换.*强化|强化券|奖励点数|技能点|.../i;
    return 数值模式.test(text) || 功能模式.test(text);
};
```

只拦截明确的数值属性和功能性机制文案，保留描述性自然语言。

#### 2.1.3 新增 `清洗描述文本` 函数

```typescript
export const 清洗描述文本 = (text: string): string
```

- 移除前缀（"这是一个"、"是"、"有一个"等）
- 移除用途说明（"可以用于"、"用于"、"用来"等）
- 清洗后为空时回退保留原文，避免 `mainSubject` 为空导致生图失败

#### 2.1.4 删除符纸符箓特殊分支

origin/main 中 `物品是否符纸符箓` 函数和相关的 `物品符箓正向约束`、`过滤符箓负面提示词`、`isTalismanPaper` 分支全部删除。

原因：新架构下符纸/符箓不再需要单独的分支处理，统一走通用逻辑即可。

#### 2.1.5 删除 `提炼中文物品描述为英文材质` 和 `构建物品描述材质提示`

这两个函数原本用于将中文物品描述翻译为英文材质提示词。新架构下云端模型直接用中文，不再需要翻译。

#### 2.1.6 删除硬编码物品名称映射（部分）

`物品名称转英文描述` 中删除了以下条目：

- `青竹法剑` / `青竹灵剑` / `灵竹剑` / `法剑`（硬编码英文名）
- `符纸`（硬编码英文描述）
- `/青竹.*剑|灵竹.*剑|竹.*法剑|法剑/` 正则匹配

#### 2.1.7 `构建物品视觉主体描述` 重构

**原逻辑**：依赖 `nameEn`（物品名称转英文描述）作为主体，`description` 作为补充。

**新逻辑**：

```typescript
const mainSubject = isLivingMount || isLivingAnimal
    ? (nameEn ? `a single real living ${qualityEn} ${isLivingMount ? 'mount animal' : 'animal'}, ${nameEn}` : `a single real living ${qualityEn} ${typeEn}`)
    : description
        ? `${description}`  // 原始描述作为主体（中英文均可）
        : (nameEn 
            ? `a single ${qualityEn} ${nameEn}` 
            : `a single ${qualityEn} ${typeEn} prop`);
```

- 优先用原始描述（已清洗），`nameEn` 降为 `category hint` 补充
- 删除 `isTalismanPaper` 分支

#### 2.1.8 `构建物品负面提示词` 简化

- 删除 `isTalismanPaper` 判断和 `过滤符箓负面提示词` 调用
- 统一返回完整负面提示词，不再对符箓做特殊过滤

#### 2.1.9 `构建物品图提示词` 调整

- 删除 `isTalismanPaper` 相关分支
- `style === '写实'` 时无条件追加 `photorealistic`（原来仅在 `renderStyle !== '写实道具'` 时追加）

#### 2.1.10 `生成物品图标` 函数 — 核心改动

##### 新增图库复用拦截（L866-927）

在生图前先查询用户图库，如果已有该物品的图片则直接复用，不再调用生图 API：

```typescript
if (!options?.force && options?.题材模式 && options?.创意工坊模块ID) {
    const galleryEntry = await 查询图库工具(item, options.题材模式, workshopModuleId);
    if (galleryEntry?.assetId || galleryEntry?.imageUrl) {
        // 复用逻辑...
    }
}
```

##### `isChineseSupported` 判断逻辑（L937）

```typescript
const isChineseSupported = /grok|imagine|gpt|image/i.test(imageApi.model || '');
```

| 匹配关键词 | 典型模型 |
|-----------|---------|
| `grok` | Grok 系列（xAI） |
| `gpt` | GPT-4o / GPT-image 系列 |
| `image` | 含 image 的模型名（如 `gpt-image-1`） |
| `imagine` | 部分想象类模型 |

##### 中文 Prompt 构建（L967-992）

```typescript
const prompt = isChineseSupported
    ? (() => {
        const 结构化描述 = structuredItem?.生图描述 || '';
        const 原始描述 = 读取文本(enrichedItem?.描述) || '';
        const 视觉描述 = 读取文本(enrichedItem?.视觉描述) || '';
        const 主体描述 = 结构化描述 || 视觉描述 || 原始描述 || `${名称}，${类型}`;
        
        const 画风约束: Record<string, string> = {
            '写实': '写实风格', '卡通': '卡通风格', '水墨': '水墨画风格',
            '像素': '像素风格', '油画': '油画风格', '素描': '素描风格',
            '国风': '中国风', '赛博朋克': '赛博朋克风格',
        };
        const 画风 = 画风约束[style] || style || '写实风格';
        return `${主体描述}，${画风}，简约背景，物品居中展示`;
    })()
    : 构建物品图提示词(enrichedItem, { 画风: style, 渲染风格: renderStyle, 来源位置: sourceLocation });
```

**优先级链**：`结构化生图描述 > 视觉描述 > 原始描述 > 名称+类型兜底`

##### 画风约束映射

仅预定义了 8 种常见画风的中文映射，未匹配的画风直接使用原值（如用户自定义画风字符串）。

##### 附加提示词处理（L995）

```typescript
const finalPrompt = extraPrompt
    ? (isChineseSupported ? `${prompt}，${extraPrompt}` : `${prompt}\n\nuser requested extra visual direction:\n${extraPrompt}`)
    : prompt;
```

中文模式下 extraPrompt 直接用逗号拼接，不包英文包装。

##### 附加正向/负面提示词（L1015-1052）

| 参数 | 中文模式 | 非中文模式 |
|------|---------|-----------|
| `附加正向提示词` | `''`（空字符串） | 完整的英文描述堆砌 |
| `附加负面提示词` | `''`（空字符串） | `构建物品负面提示词(enrichedItem)` |
| `跳过基础负面提示词` | `true` | `undefined` |

const 跳过基础负面 = options?.跳过基础负面提示词 === true;

### 2.3 `services/ai/imageGenerationDiagnostics.ts`

#### 2.3.1 代理决策系统重写

**原逻辑**：简单的 `判断可走OpenAI图片运行时代理` 正则判断。

**新逻辑**：新增 `获取代理决策` 函数，按优先级决策：

1. 用户显式开启代理 + 配了自定义代理地址 → 走自定义代理
2. 用户显式开启代理 + 未配自定义地址 → 走默认代理
3. 本地开发环境 + useRuntimeProxy + 可走同域代理 → 自动走代理
4. 在线环境 + `openai-official` 白名单 → 走 `?provider=` 模式
5. 在线环境 + 自定义端点 → 直连
6. 本地开发但代理不可用 → 直连

#### 2.3.2 新增导出函数

- `获取代理决策` — 统一代理路由决策
- `预定义供应商目标映射` — 目前仅 `openai-official` → `https://api.openai.com`

#### 2.3.3 内部函数调整

- `获取运行时代理基础地址` 新增 `自定义代理地址` 参数，优先级：用户自定义 > 当前 origin > 默认网站
- `构建OpenAI图片生成端点` 根据代理决策结果选择 `?provider=` 模式或 `?url=` 模式

## 三、冲突风险评估

### 3.1 高风险 ⚠️

| 冲突点 | 说明 |
|--------|------|
| `构建物品视觉主体描述` | 重构了主体描述构建逻辑，从 `nameEn` 优先改为 `description` 优先。如果 main 分支有其他改动依赖 `nameEn` 优先行为，会产生冲突 |
| `是否游戏机制文案` | 过滤规则从宽泛正则改为精确匹配。如果 main 分支有其他依赖原过滤行为的代码，行为会变化 |
| `构建最终图片提示词` | 删除了古代行囊纠偏逻辑。如果 main 分支有依赖此逻辑的测试或功能，会 break |
| `generateImageByPrompt` | 新增 `跳过基础负面提示词` 参数和 `代理决策` 逻辑。如果 main 分支有对 `generateImageByPrompt` 的并行修改，冲突概率高 |

### 3.2 中风险 ⚡

| 冲突点 | 说明 |
|--------|------|
| `生成物品图标` 函数签名 | 返回类型 `imageApi` 从必填改为可选，新增 `题材模式`/`创意工坊模块ID` 参数。调用方需要适配 |
| `构建图片端点` 签名 | 新增 3 个参数。如果 main 分支有调用此函数的地方，需要补充参数 |
| `获取运行时代理基础地址` 签名 | 新增可选参数，向后兼容，但调用方行为可能受新增逻辑影响 |
| 删除 `过滤符箓负面提示词` | 如果 main 分支有其他引用此函数的地方，编译错误 |

### 3.3 低风险 ✅

| 冲突点 | 说明 |
|--------|------|
| `清洗描述文本` 新增 | 纯新增函数，不影响现有代码 |
| `isChineseSupported` 判断 | 仅在 `生成物品图标` 内部使用，不影响外部 |
| 画风约束映射 | 仅在中文模式 prompt 构建中使用 |
| 图库复用拦截 | 新增逻辑，受 `options.force` / `题材模式` / `创意工坊模块ID` 三重门控 |
