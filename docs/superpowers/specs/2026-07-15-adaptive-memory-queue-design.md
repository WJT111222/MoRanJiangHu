# 后台队列自适应内存保护设计

## 目标

降低 Android WebView 和低内存设备在主剧情完成后的瞬时内存峰值，避免动态世界、规划分析、地图更新并行执行时被系统直接终止。功能阶段不得因内存保护而关闭或跳过，只允许从并行切换为串行，并向玩家说明耗时可能增加。

## 约束

- `performance.memory` 不是标准 Web API，只能作为可选信号，不能作为唯一判断依据。
- 网页端无法稳定获取系统真实剩余内存，因此必须组合 JS 堆、设备等级与当前回合数据规模。
- 判断只对当前回合生效，下回合重新评估。
- 不弹出阻塞式对话框，不改变用户已有功能开关。
- 不引入 Android 原生插件；若组合信号仍不足，再单独评估原生内存检测。

## 判断模型

新增独立的后台队列内存压力评估模块，输入以下可用信号：

1. JS 堆信号：当浏览器提供 `performance.memory` 时，读取已用堆、堆上限及使用比例。
2. 设备等级信号：当浏览器提供 `navigator.deviceMemory` 时，将低内存设备作为保守降级依据。
3. 回合负载信号：使用轻量统计估算历史正文、社交、世界、响应命令和地图数据规模，不对完整状态执行额外的 `JSON.stringify` 或深拷贝。

评估结果分为 `normal`、`high` 两档，并返回命中的原因与可诊断指标。任何可靠信号达到高风险阈值时，本回合采用串行模式；没有可用系统信号时，仍依据回合负载评估。阈值集中定义并配套单元测试，避免散落在业务流程中。

## 队列行为

- `normal`：保留当前逻辑，仅当三个阶段接口可用且渠道互不相同时并行执行。
- `high`：强制依次执行动态世界、规划分析、地图更新。
- 串行阶段之间让出主线程，及时解除上一阶段原始流式文本和临时响应的强引用。
- 最终命令合并、状态应用和自动存档流程保持原有顺序与语义。

## 用户提示

高内存压力触发时，在后台队列进度区域显示一次非阻塞提示：

> 检测到当前设备或本回合数据的内存压力较高，已自动启用节省内存模式。后台阶段将依次处理，耗时可能略有增加。

提示应复用现有阶段进度通道，不新增弹窗。若没有专用总队列提示入口，则附加到首个实际执行阶段的开始文案中，且每回合只显示一次。

## 诊断日志

每回合进入后台阶段前记录一条结构化信息，包含：

- 最终模式：并行或串行；
- 压力等级与触发原因；
- 可用的 JS 堆使用率和设备内存等级；
- 轻量回合负载统计；
- 各阶段启用状态与渠道是否允许并行。

不得把完整存档、正文、API 密钥或大型状态对象写入日志。

## 测试与验收

- 单元测试覆盖：低堆压力、堆比例过高、低内存设备、大回合负载、系统指标缺失及临界阈值。
- 队列测试覆盖：正常压力维持并行，高压力强制串行，阶段结果和最终命令顺序不变。
- 本地构建通过，现有测试不回归。
- 在移动视口或 APK 环境验证提示可见、非阻塞，且不会重复刷屏。

## 非目标

- 本次不实现 Android 原生内存插件。
- 本次不自动关闭动态世界、规划分析或地图更新。
- 本次不修改自动存档保留策略、存档格式或云同步协议。
