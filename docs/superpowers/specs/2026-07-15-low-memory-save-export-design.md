# APK 低内存存档导出设计

## 目标

修复 Android APK 在导出全部存档时因一次性加载、复制和压缩全部存档而被系统终止的问题。玩家不需要删除旧档，导出格式继续兼容现有导入流程。

## 根因

当前全部导出会通过 `getAll()` 一次性读取全部完整存档；ZIP 构建又对每条存档进行重复深拷贝，把所有图片字节保存在同一个 `files` 对象中，最后同步生成完整 ZIP 并转成 Blob。APK 写文件时还会把 Blob 转成 Base64，形成多份同时驻留的大对象。

## APK 导出全部流程

1. 读取轻量存档摘要，不读取全部完整存档。
2. 按时间顺序取得存档 ID。
3. 每次只读取一条完整存档。
4. 为该存档生成一个兼容现有格式的 ZIP。
5. 立即写入设备文档目录。
6. 释放 Blob、完整存档和临时压缩数据，再处理下一条。
7. 文件名加入序号、存档类型、时间和角色名，避免覆盖。

导出进度显示为“正在导出第 X / Y 条：存档标题”。单条失败时停止并明确报告已经成功导出的数量，避免误报全部完成。

## 网页行为

网页版继续使用现有单 ZIP 下载，避免浏览器拦截连续文件下载。本次优先修复客户反馈的 APK 闪退链路。

## 单条 ZIP 内存优化

- 删除当前重复的第二次整档深拷贝，只保留一份可修改的游戏数据副本。
- 聊天记录直接从只读原存档序列化，不再随游戏数据重复复制。
- 单条 ZIP 写入完成后不保留跨存档图片缓存。

## 测试与验收

- 单元测试证明 APK 批量导出按 ID 逐条读取、逐条生成、逐条写入，任何时刻只传递一条完整存档。
- 单元测试证明中途失败会报告成功数量和失败位置。
- 现有 ZIP 导入/导出测试继续通过。
- 生产构建通过。
- 不部署；用户确认采用后合并回 `main`。

## 非目标

- 本次不实现单个超大 ZIP 的 Android 原生流式追加写入。
- 本次不改变存档格式、图片目录结构或导入协议。
- 本次不删除、裁剪或自动清理玩家存档。
