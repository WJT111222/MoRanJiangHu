# `<judge>` 正文显示截断修复设计

## 问题

原始 AI 回复和编辑器中仍有完整正文，但聊天显示区会在 `<judge>` 出现处停止显示。当前解析器与显示层分别维护了不同的 `<judge>` 清理规则；显示层正则在找不到标准 `</judge>` 时会把标签后的全部内容删除，日志规范化流程也可能丢弃包含残留标签的整条日志。

## 目标

- 标准 `<judge>...</judge>` 只隐藏判定思考，保留判定结果和后续正文。
- 兼容模型把结束标签误写成第二个 `<judge>` 的情况。
- 兼容 `&lt;judge&gt;` 等轻量 HTML 转义形式。
- 无法确定闭合边界时优先保留玩家可见正文，不允许从 `<judge>` 删除到文本结尾。
- 保持现有 `judge_blocks` 判定卡片和原文编辑功能不变。

## 方案

建立一个可独立测试的统一 Judge 区块清理函数，由正文解析器和聊天显示防御层共同使用。

该函数按顺序扫描标签并区分以下情况：

1. 找到标准 `</judge>`：删除完整 Judge 区块并收集其内容。
2. 找到第二个 `<judge>`：将其视为兼容性闭合边界，删除该区块但保留边界后的正文。
3. 只有孤立或未闭合 `<judge>`：只删除标签本身，保留标签后的文本，避免正文截断。
4. 孤立 `</judge>`：只删除残留标签，不删除相邻正文。

`TurnItem` 不再使用“闭合标签或文本结尾”的正则清理。日志规范化仍可阻止判定思考泄漏，但不得因为一条日志包含 Judge 标签就无条件丢弃整条可见正文。

## 数据流

1. AI 原始回复进入正文解析器。
2. 统一函数提取 Judge 思考并返回清理后的正文。
3. 解析器把思考写入 `judge_blocks`，把清理后的正文解析为 `logs`。
4. 聊天显示层对历史存档或异常日志执行同一规则的防御性清理。
5. 判定结果行和 Judge 卡片按现有映射继续渲染。

## 测试

先新增失败测试，再修改生产代码：

- 标准闭合标签后正文完整保留。
- 第二个 `<judge>` 作为错误闭合时，后续正文完整保留。
- 未闭合 `<judge>` 后存在 `【判定】` 或 `【旁白】` 时，正文不被吞掉。
- HTML 转义标签得到同等处理。
- 含 Judge 残留的单条日志不会被整条删除。
- 现有判定提取、孤立 Judge 明细清理和正文解析测试继续通过。

## 非目标

- 不修改提示词中的判定协议。
- 不改变判定数值或判定卡片样式。
- 不部署、不发布、不调整版本号。
