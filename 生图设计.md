# 生图设计（调查整理）

> 调查时间：2026-07-08
> 调查范围：人物（含玩家）、物品、背景（场景）的生图调用路径，在线代理（CORS 注入）设计，以及图片的本地化、存储（图库）、远端上传、写入存档等全流程。
> 结论先行：**生图全部由客户端（浏览器 / APK）发起**；唯一的"在线代理"是项目自有的 Cloudflare Worker（`functions/` 下的 relay），只做转发 + 注入 CORS 头，**没有服务端生图业务逻辑**。当前最突出的失败是：**OpenAI 兼容端点（如 grok）在 `response_format: 'url'` 模式下，返回的图片 URL 在浏览器侧无法被直接获取/本地化**，而"测试生图界面"因为固定用 `b64_json` 走内联返回，所以不暴露该问题。大型图（NPC / 物品）必须使用 `url` 模式，因此修复方向是**修 url 模式的获取/下载路径**，而不是退回 b64_json。

## 4. 在线代理（CORS 注入）的设计与问题

### 4.1 生产 Worker：`functions/api/image-backend/openai-image-proxy/[[path]].ts`

- 强制注入 `Access-Control-Allow-Origin: *`（第 1-5 行）。
- 目标由 `provider` 参数查白名单得出，**硬编码只允许 `openai-official → https://api.openai.com`**（第 16-18、64-72 行）。
- 直接转发客户端 `Authorization` 头（用户 API Key）到上游（第 82-108 行）。
- 路径白名单：仅允许 `/v1/images/generations`、`/v1/images/edits`、`/v1/tasks/...`（第 20-26 行）。
- **不记录/不存储 Key，也不暴露用户自定义 endpoint**（目标是写死的）。

### 4.2 开发期 Vite 代理：`vite.config.ts`

- `handleOpenAiImageProxyRequest`（第 158-249 行）通过 `?url=<baseUrl>` 指定目标，并用 `isAllowedOpenAiImageProxyTarget`（第 149-156 行，仅允许 `https` + 非私网主机）做 SSRF 校验。
- 在 `imageDevProxyPlugin` 中注册为 `/api/image-backend/openai-image-proxy` 中间件（第 339-368 行）。
- **注意**：客户端只在 `当前是在线环境()` 为真时才调用该路径；本地 `npm run dev`（localhost）下该判定为假，客户端走直连，因此这个 dev 中间件在本地调试时实际不会被命中。

### 4.3 临时图片下载代理：`functions/api/image-backend/fetch-image.ts`

- 通过 `?url=` 抓取任意图片并注入 CORS 头（第 83-129 行），用于上游返回 `http://` 或跨域 `https://` 图片 URL 时避免浏览器 CORS。
- **已做 SSRF 防护**：`isPrivateHostname` + `isAllowedImageUrl`（第 19-38 行）拒绝私网/内网地址，并限制 `http(s)`、16MB。
- **生产可用，但本地 dev 不可达**（见第 15 节）。

### 4.4 其它 relay

- `functions/api/image-backend/comfyui-proxy/[[path]].ts`（仅 `cnb.run`/`cloudstudio.*`/`coding.net` 走代理）
- `functions/api/novelai/[[path]].ts`、`functions/api/pucoding-image/[[path]].ts`
- 以上均有对应的 Vite dev 镜像中间件（**fetch-image 没有**）。

## 5. 用户两个问题的确认

### 5.1 "端点和密钥泄露的可能性"

- **endpoint 泄露**：生产 OpenAI 代理白名单写死 `api.openai.com`，不暴露自定义 endpoint。真正的"任意 `?url=`"型代理是 `fetch-image`（已 SSRF 防护）和 Vite dev 代理（仅本地、`isAllowedOpenAiImageProxyTarget` 限 https 公网），低风险。
- **密钥泄露**：API Key 完全在客户端（用户填写，存于 `utils/apiConfig.ts`），由 `构建生图请求头` 以 `Authorization: Bearer <key>` 附加（`imageTasks.ts:1973-1974`）。生产 Worker 代理会经手 Key（转发给 `api.openai.com`），但第一方、同域、不记录/不转发第三方，与直连相比未新增泄露面。
- 设计隐患：① 生产 OpenAI 代理是无鉴权"开放式转发器"，可被限额洗白，建议加来源校验/签名；② 若要彻底规避 Worker 经手 Key，需服务端代持 Key + 用户令牌授权。

### 5.2 "本地调试时候无法访问"

- 根因：`当前是在线环境()`（`imageGenerationDiagnostics.ts:202-206`）硬编码只认两个生产域名。本地下 `useRuntimeProxy` 分支被跳过 → 走 `构建直连端点`；多数 OpenAI 兼容上游不回传 CORS → 浏览器拦截。
- Vite dev 代理中间件虽存在，但客户端本地不会路由到它 → "死路"。
- 更重要的是：**`/api/image-backend/fetch-image` 这个图片下载代理在 Vite dev 完全没注册**（见第 15 节），这是本地无法调试 `url` 模式生图的直接技术原因。

## 6. OpenAI 兼容端点生图失败的根因（重点）

1. `供应商ID` 仅在 `baseUrl` 命中 `api.openai.com` 时为 `openai-official`，否则为 `openai-custom`（`imageTasks.ts:4630`）。
2. `openai-custom` 不在白名单，代理分支跳过。
3. 请求改为 `构建直连端点`（`imageGenerationDiagnostics.ts:208-229`），直连自定义 endpoint。
4. 直连能否成功取决于上游是否回传 `Access-Control-Allow-Origin`。
5. 即使"在线环境"，自定义兼容端点也用不了代理；本地则根本不进入代理分支。

**但本项目的真实失败（见第 8 节）并非"请求被 CORS 拦"，而是 `url` 模式返回的图片 URL 在浏览器侧无法被获取/本地化。** 这两者要区分清楚：请求能发出去（所以测试能连），但 `url` 模式的"取回图片"这一步在浏览器会失败。

## 7. url 模式：返回图片的获取 / 本地化（核心修复区）

> **重要前提**：NPC / 物品都是大图，`response_format` **必须用 `url`**（`b64_json` 承载不了大图，不能退回 b64）。所以修复方向是"让 `url` 模式的返回值能被正确取回并本地化"，而不是改为 b64。

### 7.1 两条路径的 response_format 差异

| 维度 | 测试生图界面 | 实际生图 |
|---|---|---|
| 调用函数 | `测试OpenAI兼容图片接口`（`services/ai/openaiImageTest.ts`） | `generateImageByPrompt`（`services/ai/imageTasks.ts:4624`） |
| `response_format` | 非 GPT 模型固定 `b64_json`（`openaiImageTest.ts:104`：`params.responseFormat \|\| 'b64_json'`） | 来自 `apiConfig.图片响应格式`，**默认 `'url'`**（`apiConfig.ts:2338` 默认分支） |
| 返回形式 | 内联 base64，直接预览，**无二次跨域** | 服务端返回图片 URL（或把请求 3xx 重定向到图片地址） |
| 取回方式 | 不需要 | 需要再次 `fetch` 该 URL 并本地化为 blob |

非 GPT 模型实际请求体（`imageTasks.ts:4749-4750`）：

```ts
if (!isGptImageModel && !isChatCompletionsEndpoint) {
  requestBody.response_format = responseFormat;   // 默认 'url'
}
```

### 7.2 失败机理（用户补充的关键信息）

- 该 grok 兼容端点**本身也会对图片做 CORS 代理**：原图其实是 Grok 的 url，但端点会返回一个可被浏览器取回的地址。"直连没必要 CDN"——即返回的图片地址应可直接取，而不必绕 CDN。
- 但 `generateImageByPrompt` 内的生图 `fetch`（`imageTasks.ts:4916`）**未设置 `redirect` 选项（默认 `follow`）**。当 `url` 模式让端点/上游以 **3xx 重定向到图片地址**（或返回一个跨域图片 URL）时：
  - 浏览器跟随重定向到无 CORS 头的外部主机 → 抛 `TypeError: Failed to fetch`，错误归到发起 `fetch` 的那一行（`imageTasks.ts:4916`，构建映射 `:3830`）。
  - 这正是 `运行报错.log` 的现象：endpoint 直连可达（测试能连），但 `url` 模式生图请求在浏览器侧失败。
- origin/main 的 APK 之所以能用 `url` 正常工作，是因为：APK 走 **CapacitorHttp（原生 HTTP，不强制 CORS）**，且对返回的 url 走**作者的图片代理**取回。浏览器侧缺少这两点，因此失败。

### 7.3 验证建议

- 浏览器 Network 面板观察：实际生图失败时，是否有一次 `images/generations` 请求返回 3xx 重定向到无 CORS 头的地址（或返回 JSON 里带一个跨域 url）。
- 或临时把 `图片响应格式` 设为 `b64_json` 验证"内联即成功、url 即失败"的差异（仅用于确认机理，**不作为最终修复**）。

### 7.4 正确修复方向（保留 url，修取回路径）

1. **生成请求手动处理重定向**：在 `generateImageByPrompt` 的 `fetch`（`:4916`）加 `redirect: 'manual'`。若响应为 3xx，读取 `Location` 头作为图片地址（或再次请求该地址），避免在浏览器侧自动跟随到无 CORS 的主机。
2. **返回的 url 一律走第一方图片代理取回**：在 `persistImageAssetLocally`（`:5134-5166`）中，对跨域 `https://` URL **主动**经 `构建临时图片下载代理地址`（fetch-image 代理）下载，而不是先直连、失败再回退。这与 APK/作者原版行为一致，且代理会注入 CORS 头、做 SSRF 防护。
3. **让 fetch-image 代理在所有环境可达**（见第 15 节）：当前 `构建临时图片下载代理地址` 用 `buildSyncApiUrl('/api/image-backend/fetch-image')`（`imageTasks.ts:169`），当 `VITE_SYNC_API_BASE_URL` 未设时返回**相对路径**；而 Vite dev 没有注册该中间件 → 本地拿到 SPA 的 `index.html` → `校验下载结果是图片`（`imageTasks.ts:5010-5043`）抛"不是有效图片"。需在 `vite.config.ts` 注册 `/api/image-backend/fetch-image` 中间件（转发到部署的 Worker 或同域代理），并放宽"仅在线环境"的硬编码限制。
4. **统一测试与生产**：把测试界面的 `response_format` 逻辑与生产对齐（或测试也覆盖 `url` 模式），消除"测试通过、生产失败"的假阳性。

## 9. 图片存储层（IndexedDB）

- 存储名 `'image_assets'`（`dbService.ts:49`，创建于 `:828-829`），记录结构 `{ id, dataUrl, createdAt }`（`写入图片资源记录` `:611-625`）。
- 引用格式 `wuxia-asset://<id>`（`utils/imageAssets.ts:13`），由 `创建图片资源引用`(`:139-142`) 生成，`解析图片资源引用ID`(`:144-147`) 解析。
- 保存函数 `保存图片资源`（`dbService.ts:840-881`）：
  - `是远程地址(dataUrl)` 时直接原样返回（不落本地，`utils/imageAssets.ts`/`dbService.ts:845-846`）；
  - 否则写入 `image_assets` 并 `注册图片资源缓存`，返回 `wuxia-asset://` 引用；
  - 按签名去重（`图片资源签名缓存` `dbService.ts:66`，命中返回已有引用 `:852-865`）；
  - 推入 `延迟上传队列`（`:878`）做异步 R2 上传。
- `保存图片资源并返回同步地址`（`dbService.ts:883-885`）会同步触发 `上传图片资源到图床并登记`。
- **无 IndexedDB 配额处理**：`QuotaExceededError` 直接 reject，向上冒泡 → 图片静默未保存。
- 内存 LRU 缓存上限 80 条 / 42MB（`utils/imageAssets.ts:15-16`），仅限内存，存储本身无上限。

## 10. 图片本地化（persistImageAssetLocally）与下载代理

- `persistImageAssetLocally`（`imageTasks.ts:5045-5167`）：把 `generateImageByPrompt` 返回的 `图片URL` / `本地路径` 归一化为本地 `wuxia-asset://` 引用。成功后 `图片URL` 清空、`本地路径` 置为引用（`:5162-5166`）。
- 对 `http(s)` URL 的下载逻辑（`:5134-5166`）：
  - `shouldUseTemporaryImageProxy = /^http:\/\//i.test(imageUrl) && !是否本机或局域网图片地址(imageUrl)`；

  - 对 `http://` 主动走代理；对 `https://` 先直连，失败再回退代理（`:5140-5141`）。
- **问题**：`buildSyncApiUrl` 在 `VITE_SYNC_API_BASE_URL` 未设时返回**相对路径** `/api/image-backend/fetch-image`；而 Vite dev 未注册该中间件 → 返回 SPA `index.html` → `校验下载结果是图片`（`:5010-5043`）抛"不是有效图片"。这是"生图结果无法保存 / 本地无法调试"的直接原因（详见第 15 节）。

## 12. 远端上传 / 同步（R2）与上传失败

- 上传函数 `上传DataUrl到图床`（`services/imageHostService.ts:261-422`）：端点 `/api/image-host/upload`，由 `buildImageHostProxyUrl`（`:42-49`）构建（未设 `VITE_SYNC_API_BASE_URL` 时在 web 走相对路径）。重试最多 3 次（backoff），最终失败抛 `图床上传失败`（`:385`）。
- 持久层调用 `上传图片资源到图床并登记`（`dbService.ts:427-450`）：
  - 云端游玩模式 / 签名冷却命中时跳过；
  - **best-effort**：上传失败仅 `recordDiagnosticLog('warn', '图片资源已保存本地，图床上传稍后重试')`，返回 `''`，**不阻断本地保存**。
- 延迟队列 `执行延迟上传队列`（`dbService.ts:87-101`，`App.tsx:2728` 回首页时触发）：**错误被吞掉**（`:95-97`）。
- 按签名的冷却 `是否跳过图床上传` / `标记图床上传失败跳过`（`dbService.ts:559-576`，存 `localStorage`）：连续失败后该签名在 TTL 内被永久跳过上传。
- `requiresRemoteSyncApi`（`nativeRuntime.ts:52-58`）：仅原生 Capacitor 在 localhost/127.0.0.1 时为 true，且**必须**配置 `VITE_SYNC_API_BASE_URL`。

> 结论：远端上传失败**本身不会**阻止本地保存（设计如此）；但连续失败后的冷却 + 静默吞错，会让用户看到"上传失败"且无明显自愈，给人一种"图库状态异常/上传失败"的观感。

## 13. 生图结果写入存档（外置化图片字段）

**调用链（NPC 为例）**：
1. `generateImageByPrompt` → `{ 本地路径?, 图片URL? }`。
2. `persistImageAssetLocally`（`:5045`）→ `dbService.保存图片资源` 写入 `image_assets`，返回 `wuxia-asset://` 引用。
3. 工作流把引用写入 React 游戏状态：`npcImageWorkflow.ts:699` → `:722` `更新NPC最近生图结果`，落到 `图片档案.最近生图结果` / `生图历史`（`:724-743`），头像/立绘槽位 `:747-748`。
4. **存档时** `外置化图片字段`（`dbService.ts:959-1001`）遍历存档，把特定 key 下的 **原始 dataUrl** 转成引用；从存档写入处 `:1983 / :2106 / :2937 / :2950` 调用。

**其它入口**：`npcSecretImageWorkflow.ts:414`、`npcImageStateWorkflow.ts:948`、`sceneImageWorkflow.ts:330`、`sceneImageArchiveWorkflow.ts:227`、`itemImageGeneration.ts:1036`（含图库复用 `:869-924`）、玩家头像独立路径 `saveArchiveService.ts:127-145`。

**引用丢失 / 未落盘的位置**：
1. **`persistImageAssetLocally` 在状态写入前抛错**：如下载步骤失败（fetch-image 不可达，见第 10/15 节），工作流 `catch`（`npcImageWorkflow.ts:700-703`）重抛 → `更新NPC最近生图结果`（`:722`）不执行 → 引用未写入状态 → **图片进不了存档**（直接对应"生图结果无法保存到存档"）。
2. **异步快照竞态**：资源在步骤 2 写入 `image_assets`，但存档只在真正保存/自动保存时经 `外置化图片字段` 才含引用（步骤 4）。若用户在自动保存前刷新，内存引用丢失，`image_assets` 中变成**孤儿**。无"生图即写存档"的同步路径。
3. **key 名依赖**：`外置化图片字段` 只在固定 key 允许列表（`本地路径` / `图片URL` / `背景图片` / `头像图片URL` / `*图片URL` / `*音频URL`）下转换 raw dataUrl；其它 key 下的 raw dataUrl 会原样内嵌进存档（膨胀且不被注册），引用则原样保留（安全）。
4. **远程裸 URL**：若 `generateImageByPrompt` 返回 `https://` `图片URL` 且绕过了 `persistImageAssetLocally`，存档只存裸远程 URL、无本地副本、无 `image_assets` 条目，远端失效即图丢。

## 14. 孤儿图片 / 缺失引用 / 迁移

- **孤儿检测（仅报告）**：`诊断本地数据`（`dbService.ts:3322+`，`:3340-3355`）算 `orphanIds`（在 `image_assets` 但未被引用）与 `missingIds`（被引用但资源缺失），仅作为警告，**不自动修复**。
- **孤儿清理（手动触发）**：`清理未引用图片资源`（`dbService.ts:1837-1868`）删除所有未被 `referencedIds` 引用的 `image_assets` id。`referencedIds` 由 `读取存档设置图片引用快照`（`:1055-1120`）构建，覆盖：所有存档（`STORE_NAME`）、设置（`SETTINGS_STORE`）、`USER_IMAGE_GALLERY_STORE`（`:1096-1115`，保护图库 `assetId`/`imageUrl`）、以及远程兜底映射。清理只在显式调用时执行（多处 `:2711 / :2738 / :2757 / :2799` 等），孤儿会累积。
- **旧存档迁移**：`迁移图片资源到独立存储`（`dbService.ts:2907-2963`）遍历所有存档+设置跑 `外置化图片字段` 写回；非事务、逐条 `put`，中途崩溃可能留下半外置存档；仅在末尾置 `图片资源迁移版本键`（`:2961`）。

## 15. 本地无法调试生图（fetch-image 代理在 Vite dev 缺注册）

- 客户端 `构建临时图片下载代理地址`（`imageTasks.ts:169-173`）→ `buildSyncApiUrl('/api/image-backend/fetch-image')`（`nativeRuntime.ts:12-16`），`VITE_SYNC_API_BASE_URL` 未设时返回**相对路径**。
- `vite.config.ts` 的 `imageDevProxyPlugin` 只注册 `/api/novelai`、`/api/pucoding-image`、`/api/image-backend/openai-image-proxy`、`/api/image-backend/comfyui-proxy`（`:339-369`）；**`/api/image-backend/fetch-image` 未注册**，也没有 `server.proxy` 块（`:387-403`）。
- **后果**：本地 `fetch('/api/image-backend/fetch-image?url=...')` 命中 Vite → 返回 SPA `index.html`（HTTP 200，`text/html`）。`response.ok` 为真，但 `校验下载结果是图片`（`:5010-5043`）抛"下载结果不是有效图片"。
  - 对 `http://` 返回图：代理**必需**却不可达 → `persistImageAssetLocally` 必抛 → 图片不落 `image_assets`、不进存档（"生图无法保存"+无法调试）。
  - 对 `https://`：先直连（CORS 允许则成；CORS 拦截则回退到同一不可达代理 → 也失败）。
- **生产正常**：`functions/api/image-backend/fetch-image.ts` 存在且带 SSRF 防护，从 Worker 提供。破损只发生在**本地 `vite` dev** 与"原生 Capacitor 且 `VITE_SYNC_API_BASE_URL` 未设"时。
- **修复**：在 `vite.config.ts` 注册 `/api/image-backend/fetch-image` 中间件（转发到部署的 Worker 或同域代理，复用 `fetch-image.ts` 的 SSRF 校验）；或加 `server.proxy` 把 `/api` 指到 Worker。本地调试即可正常取回 `url` 图片。

## 17. 后续改造方向（按优先级，待确认未实施）

1. **修 url 模式取回（最高优先）**：生成 `fetch` 加 `redirect: 'manual'`（`:4916`）；`persistImageAssetLocally` 对跨域 `https://` 主动走 fetch-image 代理（`:5134`）；让 fetch-image 代理在本地 dev 可达（vite 注册，第 15 节）。对齐 APK/作者原版行为。
2. **统一测试与生产 response_format**：测试界面覆盖 `url` 模式或对齐生产逻辑，消除"测试通过、生产失败"。
3. **放宽代理可达性**：把 `当前是在线环境()` 改为"是否部署在可被 Worker 代理的同源环境"，或本地默认走 Vite 代理，去掉硬编码两域名限制。
4. **存储健壮性**：`写入图片资源记录` 捕获 `QuotaExceededError` 并降级（如压缩/清理最旧）；避免大图静默丢失。
5. **图库状态自愈**：`ImageManagerModal` 渲染时校验 `assetId` 有效性；孤儿图库条目在加载时批量清理，而非仅惰性删除。
6. **存档竞态**：考虑在 `persistImageAssetLocally` 成功后立即把引用同步进当前存档草稿，减少"刷新即孤儿"。
7. **放宽代理白名单（谨慎）**：允许受信自定义 OpenAI 兼容 endpoint 经代理（Worker 侧 SSRF/合法性校验 + 来源限制 + 请求签名），避免开放式转发器滥用。
8. **密钥代持（彻底规避 Worker 经手 Key）**：服务端代持 Key + 用户令牌授权，当前 `wrangler.jsonc` 未存生图 Key，需新增密钥管理与授权流程。
